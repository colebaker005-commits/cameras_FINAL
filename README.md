# DC Traffic Camera Impact Analysis

A difference-in-differences analysis of every active DC automated traffic enforcement camera, measuring whether crashes within two blocks of each camera fell in the year after installation relative to the citywide trend.

## What's in this repo

- `scripts/` — the pipeline, one script per step (01 through 06). Runs end-to-end.
- `notebooks/walkthrough.ipynb` — a narrated walkthrough of the same analysis, readable top-to-bottom on GitHub.
- `data/raw/` — the source Excel workbook (public DDOT crash and camera data).
- `outputs/` — where results land (Excel workbook, figures). Regenerated by the scripts.

## Data sources

- **Crashes in DC** — [DDOT open data](https://opendata.dc.gov/datasets/crashes-in-dc), filtered to 2021–2026.
- **Automated Traffic Enforcement Cameras** — [DDOT open data](https://opendata.dc.gov/datasets/automated-traffic-enforcement), all camera locations with install dates.

The raw file in `data/raw/Midterm_Memo.xlsx` is a snapshot of both datasets combined into one workbook for this analysis.

## Method in one paragraph

For each of 283 eligible cameras, the analysis counts crashes within a 200-meter buffer in the 365 days before installation and in the 365 days after. The same counts are computed citywide for the same windows. The difference-in-differences (DiD) metric is `(zone % change) − (citywide % change)` — a negative DiD means the zone did better than the citywide trend. Three validity checks run alongside: a placebo (same analysis with the install date shifted back one year), a sensitivity check (150m, 200m, 250m buffers), and a mechanism check (for speed cameras, speeding-involved crashes vs. non-speeding).

Full methodology is in the Methodology tab of `outputs/camera_analysis_results.xlsx` and in `notebooks/walkthrough.ipynb`.

## Running it yourself

### Prerequisites

- Python 3.10 or newer
- About 500 MB of free disk space (the raw workbook is 33 MB and the pipeline generates intermediate files)

### Setup

```bash
# Clone the repo
git clone https://github.com/YOUR_USERNAME/dc-camera-analysis.git
cd dc-camera-analysis

# Create a virtual environment (recommended)
python3 -m venv venv
source venv/bin/activate    # on Windows: venv\Scripts\activate

# Install Python packages
pip install -r requirements.txt
```

### Run the pipeline

The scripts are numbered so they run in order. You can run them one at a time:

```bash
python scripts/01_clean_crashes.py
python scripts/02_filter_cameras.py
python scripts/03_run_analysis.py
python scripts/04_placebo_test.py
python scripts/05_make_figures.py
python scripts/06_build_workbook.py
```

Or all at once:

```bash
bash run_pipeline.sh
```

Each script prints what it's doing and what it wrote. Script 03 is the slowest (~20 seconds); everything else runs in a few seconds.

## Limitations

This is exploratory journalism analysis, not peer-reviewed research. Key caveats:

- Cameras are not randomly placed. They're installed where crashes are elevated, creating regression-to-the-mean risk. The placebo test detects but does not fully eliminate this.
- The 200m buffer is a straight-line Euclidean proxy for "two blocks," not a street-network distance.
- Camera installations often coincide with other streetscape changes (signage, paint, bollards). Effects cannot be attributed to the camera alone without on-the-ground verification.
- Fatal and serious-injury per-camera-type counts are too sparse to support confident claims.

See the Methodology section of the PDF briefing for the full list.
